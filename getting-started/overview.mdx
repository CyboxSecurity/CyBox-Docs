# Start here

CyBox is a security control layer for AI-generated code and fast-moving software exposure.

It gives teams a simple operating model:

<Steps>
  <Step title="Connect assets">
    Bring in repositories and public web assets so CyBox can understand what your team is building and exposing.
  </Step>
  <Step title="Surface risk">
    CyBox scans code, dependencies, secrets, infrastructure definitions, and live internet-facing assets.
  </Step>
  <Step title="Take action">
    Teams triage findings, generate fix pull requests, and enforce policies directly in the pull request flow.
  </Step>
</Steps>

## Who CyBox is built for

- **Security teams** that need control without slowing development down
- **Engineering leaders** that want fast feedback inside the normal delivery flow
- **Founders and CTOs** that need visibility into what AI tools are producing across the company

## What makes CyBox different

Traditional security tools usually stop at detection.

CyBox is built to help teams move from **finding** to **decision**:

- It connects findings to the pull request that introduced them
- It helps teams remediate faster with guided fixes and AutoFix
- It monitors live assets, not just source code
- It adds governance in the place where shipping decisions happen: the PR

## The core CyBox workflow

1. Connect GitHub and add your first repository
2. Run a baseline scan
3. Review findings by severity and asset
4. Enable PR Firewall for merge-time control
5. Add public websites or domains for live exposure validation

<Info>
A good first rollout is one critical repository, one public asset, and one team. That keeps setup simple and gives you quick value.
</Info>
