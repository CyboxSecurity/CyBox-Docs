# Security model

CyBox is designed for multi-tenant use with organization-scoped access and isolated customer data.

## Tenant isolation

- Every organization is logically separated
- Requests are validated against organization membership
- Data access is scoped to the active organization context
- Isolation is enforced at the storage and application layers

## Source code handling

CyBox is designed to minimize code persistence during scanning.

<AccordionGroup>
  <Accordion title="Repository content">
    Repository data is used for scanning workflows and is not intended to become long-term product content. Teams primarily work from findings, metadata, and review results.
  </Accordion>
  <Accordion title="Findings data">
    Findings include the context needed to understand and remediate an issue — location, severity, evidence, and status.
  </Accordion>
  <Accordion title="Secrets exposure">
    When credentials are detected, CyBox preserves only the minimum evidence needed for triage and verification. Full secret values are not stored.
  </Accordion>
  <Accordion title="Access tokens">
    Connected integration credentials are scoped to the minimum access required for the feature being used.
  </Accordion>
</AccordionGroup>

## Practical controls

| Area | Control |
|---|---|
| **Organization access** | Membership-based access to assets and findings |
| **Sensitive data** | Minimized retention and least-privilege handling |
| **PR decisions** | Logged verdicts for auditability |
| **Integrations** | Explicit customer authorization required |
