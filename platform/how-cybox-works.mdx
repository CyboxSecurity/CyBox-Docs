# How CyBox works

CyBox follows a clear operating flow: connect assets, scan for risk, prioritize findings, and enforce decisions where code gets merged.

## End-to-end flow

```text
Assets connected
   ↓
Scans run across code and live assets
   ↓
Findings are deduplicated and prioritized
   ↓
PR Firewall evaluates introduced risk in pull requests
   ↓
Teams remediate manually or with AutoFix
```

## 1. Assets are connected

CyBox works with repositories and public web assets.

Typical starting point:

- One or more connected GitHub repositories
- One public website, preview deployment, or domain

## 2. Scans run

CyBox reviews connected assets for:

- Code vulnerabilities and insecure patterns
- Dependency risk
- Exposed credentials and secrets
- Infrastructure misconfigurations
- License risk
- Live internet-facing exposure on web assets

## 3. Findings are normalized

CyBox groups repeated detections so the same issue is not treated as new on every scan.

Each finding keeps the context teams need to act:

- Severity
- Asset
- Location or endpoint
- Evidence
- Suggested remediation
- Status over time

## 4. PR Firewall makes merge-time decisions

For pull requests, CyBox focuses on **introduced risk** rather than historic issues already sitting in the codebase.

That lets teams answer the most important question quickly:

**Should this change be allowed, warned on, or blocked?**

## 5. Teams remediate and move forward

From the finding or review flow, teams can:

- Assign or discuss remediation
- Ignore with reason where appropriate
- Generate an AutoFix PR for supported findings
- Re-scan to confirm resolution

<Note>
CyBox is most effective when baseline issues are reviewed once, and PR Firewall is then used to keep new risk from slipping through.
</Note>
