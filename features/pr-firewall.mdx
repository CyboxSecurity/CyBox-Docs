# PR Firewall

PR Firewall puts security governance directly in the pull request flow.

Instead of discovering issues after merge, teams can make a decision while the change is still under review.

## What PR Firewall does

For each pull request, CyBox evaluates the risk introduced by that change and returns a verdict:

| Verdict | Meaning |
|---|---|
| **Allow** | The pull request can move forward |
| **Warn** | The pull request should be reviewed carefully before merge |
| **Block** | The pull request introduces risk that should be fixed before merge |

## Why that matters

PR Firewall helps teams answer three important questions fast:

1. **Did this PR introduce meaningful risk?**
2. **Is the risk severe enough to stop merge?**
3. **What exactly needs to be fixed?**

## What CyBox evaluates

PR Firewall is designed to focus on issues introduced by the pull request, so teams do not get blocked by old baseline problems that already existed.

That keeps reviews practical and reduces friction.

## AI-aware review context

CyBox can surface signals that a pull request may have been created or heavily influenced by AI tooling.

That context helps teams understand where additional scrutiny may be useful, while verdicts remain grounded in the risk actually found.

## Typical rollout

<Steps>
  <Step title="Install GitHub integration">
    Connect the repository through the CyBox GitHub integration.
  </Step>
  <Step title="Review baseline findings">
    Understand the current state of the repository so old issues do not confuse rollout.
  </Step>
  <Step title="Enable branch protection">
    Require the CyBox check if you want Block verdicts to stop merge in GitHub.
  </Step>
  <Step title="Monitor and tune">
    Review verdict trends, override cases, and team feedback as usage grows.
  </Step>
</Steps>

## What teams see in CyBox

In the PR Firewall view, teams can review:

- Pull request verdict
- Introduced findings
- Severity distribution
- Linked repository and PR
- Review history and decision context

## Good use cases

- Preventing new secrets from being merged
- Catching risky auth or permission changes before release
- Highlighting dangerous dependency or infrastructure changes in the PR itself
- Adding governance without forcing teams into a separate workflow
