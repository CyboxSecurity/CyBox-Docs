## Where it sits

Most teams have no control layer between AI-generated code and production. A developer accepts a Copilot suggestion, opens a PR, it merges, it ships. PR Firewall closes that gap.

It sits **between the pull request and the merge button** ‚Äî specifically between the GitHub check system and your branch protection rules. Every PR is scanned, evaluated against your rules, and receives a verdict before anyone can merge.

```
Developer opens PR
        ‚Üì
  GitHub webhook
        ‚Üì
  CyBox scans the diff
  (SAST ¬∑ Secrets ¬∑ SCA ¬∑ AI detection)

  Rules engine evaluates
  (top-to-bottom, first match wins)
        ‚Üì
  Allow / Warn / Block
        ‚Üì
  GitHub Check posted on PR
        ‚Üì
  Block ‚Üí merge prevented
  Allow/Warn ‚Üí merge proceeds
        ‚Üì
  main branch ‚Üí Production
```

## Verdicts

Rules are evaluated top-to-bottom. The **first rule that matches** determines the verdict. If no rule matches, the PR is Allowed by default.

| Verdict | GitHub Check | Effect |
|---------|-------------|--------|
| ‚úÖ **Allow** | Passes | PR can merge ‚Äî no action required |
| ‚öÝÔ∏è **Warn** | Passes with warning | Developer is notified; merge is still allowed |
| üö´ **Block** | Fails | Merge is prevented until findings are resolved or an admin overrides |

<Note>
If the CyBox scan itself fails (timeout, infra error), a neutral check is posted and the PR is **not** blocked. A scan failure is never treated as a security verdict.
</Note>

## Firewall Rules

Rules live at **PR Firewall ‚Üí Firewall Rules**. They define what to look for in a PR and what to do when it matches.

Rules are evaluated **top-to-bottom ‚Äî first match wins**. Drag the grip handle (‚Ýø) on any rule row to reorder priority.

### Rule anatomy

Every rule has four parts:

**Name** ‚Äî shown in the PR Firewall events table (e.g. "Block auth changes").

**Developer Message** ‚Äî the message posted directly in the GitHub PR review when the rule matches. Write something actionable: `Protected authentication area ‚Äî security review required` is better than `Rule matched`.

**Scope** ‚Äî which repositories the rule applies to. Defaults to all connected repositories. Narrow to specific repos for environment-specific policies (e.g. stricter rules for your production API repo).

**Status** ‚Äî toggle a rule on or off without deleting it. Useful for testing or temporarily disabling rules.

---

### Conditions (WHEN)

Conditions define what must be true about a PR for the rule to fire. All conditions in a rule are combined with **AND** ‚Äî every condition must match. Multiple values within a single condition use **OR**.

Click **+ Add condition** to choose from:

<AccordionGroup>
  <Accordion title="Path ‚Äî match changed file paths">
    Triggers when the PR includes changes to files under a specific path pattern.

    **Operator:** `matches`
    **Value:** A glob pattern ‚Äî e.g. `/auth/**`, `/api/payments/**`, `src/config/**`

    This is the most precise way to protect sensitive directories. The rule fires whenever any file under the pattern is modified in the diff.

    **Examples:**
    - `path: /auth/**` ‚Üí Block ‚Äî require security review for any auth code change
    - `path: /migrations/**` ‚Üí Block ‚Äî protect database migration files
    - `path: /config/**` ‚Üí Warn ‚Äî flag configuration changes for review
  </Accordion>

  <Accordion title="File ‚Äî match specific filenames">
    Triggers when the PR includes changes to a specific file or filename pattern.

    **Operator:** `matches`
    **Value:** A filename or glob ‚Äî e.g. `*.env`, `Dockerfile`, `secrets.json`

    More targeted than Path ‚Äî use this for individual files rather than whole directories.

    **Examples:**
    - `file: *.env` ‚Üí Block ‚Äî catch accidental `.env` file commits
    - `file: package-lock.json` ‚Üí Warn ‚Äî flag any lockfile changes
    - `file: Dockerfile` ‚Üí Warn ‚Äî review container changes before merge
  </Accordion>

  <Accordion title="Code area ‚Äî match by functional area">
    Triggers based on the functional area of code being changed, detected automatically by CyBox from the diff ‚Äî no path configuration needed.

    **Values:** `infrastructure`, `authentication`, `payments`, `database`, and others.

    CyBox classifies the code area from the content of the diff, not just the file path. A payment function added to a utility file will still trigger a `payments` rule.

    **Examples:**
    - `code area: authentication` ‚Üí Block ‚Äî enforce review for any auth logic change
    - `code area: infrastructure` ‚Üí Warn ‚Äî flag infrastructure changes
    - `code area: payments` ‚Üí Block ‚Äî require sign-off on payment code
  </Accordion>

  <Accordion title="Severity ‚Äî match by finding severity">
    Triggers when CyBox detected one or more findings at or above a specific severity level in the PR diff.

    **Values:** Critical, High, Medium, Low

    Only evaluates **introduced findings** ‚Äî findings mapped to the diff, not pre-existing issues in the codebase.

    **Examples:**
    - `severity: Critical` ‚Üí Block ‚Äî never let a Critical finding merge
    - `severity: High` ‚Üí Block ‚Äî block High and above
    - `severity: Medium` ‚Üí Warn ‚Äî surface Medium findings without blocking
  </Accordion>

  <Accordion title="AI involvement ‚Äî match AI-generated PRs">
    Triggers when CyBox detected that the PR was created or significantly contributed by an AI coding tool.

    **Values:** `present` / `not present`

    Detection uses a multi-signal system:
    - Copilot co-author metadata in commit messages
    - AI tool config files present in the repo (`.cursor/`, `CLAUDE.md`, `.windsurfrules`, `copilot-instructions.md`)
    - AI-assisted label on the PR
    - Bot-authored PRs from v0, Lovable, Bolt, Devin, and similar tools

    **Examples:**
    - `AI involvement: present` ‚Üí Warn ‚Äî require human review on all AI-generated PRs
    - `AI involvement: present` + `severity: High` ‚Üí Block ‚Äî block high-risk AI PRs automatically
  </Accordion>

  <Accordion title="Secret detected ‚Äî match exposed credentials">
    Triggers when CyBox's secrets scanner (Gitleaks) found an exposed credential, API key, token, or private key anywhere in the PR diff.

    **Values:** `Yes` / `No`

    This is the single most critical rule to have active. Secrets committed to code ‚Äî even briefly ‚Äî are compromised and must be rotated.

    **Example:**
    - `secret detected: Yes` ‚Üí Block ‚Äî unconditionally block any PR with an exposed secret
  </Accordion>

  <Accordion title="Trust boundary risk ‚Äî match cross-boundary code">
    Triggers when CyBox detected a trust boundary violation ‚Äî code where untrusted input flows toward a privileged operation without proper validation or sanitization.

    **Values:** `Yes` / `No`

    **Example:**
    - `trust boundary risk: Yes` ‚Üí Block ‚Äî require security review for any trust boundary violation
  </Accordion>

  <Accordion title="Dependency change risk ‚Äî match supply chain changes">
    Triggers when the PR introduces changes to dependencies ‚Äî new packages added, versions upgraded, or packages removed.

    **Values:** `Yes` / `No`

    Use this to enforce review of supply chain changes regardless of whether a CVE is currently known.

    **Examples:**
    - `dependency change risk: Yes` ‚Üí Warn ‚Äî flag all dependency changes for review
    - `dependency change risk: Yes` + `severity: High` ‚Üí Block ‚Äî block dependency changes that introduce known CVEs
  </Accordion>

  <Accordion title="Documentation-only PR ‚Äî match docs-only changes">
    Triggers when every changed file in the PR is a documentation file (`.md`, `.mdx`, `.txt`, etc.) with no code changes.

    **Values:** `Yes` / `No`

    Use this to fast-track documentation PRs past stricter security rules, or to explicitly Allow them so they bypass Block rules lower in the stack.

    **Example:**
    - `documentation-only PR: Yes` ‚Üí Allow ‚Äî exempt docs PRs from all Block rules
  </Accordion>
</AccordionGroup>

### Actions (THEN)

Once conditions match, the rule applies one of three actions:

| Action | Effect |
|--------|--------|
| **Allow** | Explicitly pass this PR ‚Äî useful for exempting safe patterns before broader Block rules below |
| **Warn** | Post a warning on the PR ‚Äî visible in the GitHub check, but does not block merge |
| **Block** | Fail the GitHub Check ‚Äî PR cannot merge until the issue is resolved or a repo admin overrides |

## Creating a rule

1. Go to **PR Firewall ‚Üí Firewall Rules ‚Üí + Add Rule**
2. Enter a **Name** and a **Developer Message**
3. Set the **Scope** (default: all repositories)
4. Click **+ Add condition** and configure one or more conditions
5. Select the **Action** ‚Äî Allow, Warn, or Block
6. Save ‚Äî the rule activates immediately

---

## Rule priority and ordering

Rules are evaluated **top to bottom**. The first rule that matches wins ‚Äî everything below it is ignored for that PR.

Use the drag handle (‚Ýø) to reorder. Recommended ordering:

1. **Explicit Allow rules first** ‚Äî e.g. docs-only PRs that should skip all checks
2. **High-priority Block rules** ‚Äî secrets detected, protected paths
3. **Broader Warn rules** ‚Äî AI involvement, dependency changes, infra code
4. **Catch-all rules last** ‚Äî e.g. Warn on any PR with Medium or above findings

## Setup

### 1. Install the CyBox GitHub App

See [GitHub Integration](/integrations/github) for full setup instructions.

### 2. Enable branch protection

For Block verdicts to prevent merging:

1. Go to your repo ‚Üí **Settings ‚Üí Branches ‚Üí Add branch protection rule**
2. Set branch name pattern to `main` (or your default branch)
3. Enable **Require status checks to pass before merging**
4. Search for and select **CyBox Security Scan**
5. Save

### 3. Configure your rules

Go to **PR Firewall ‚Üí Firewall Rules ‚Üí + Add Rule** and build out the rules that reflect your team's policy.

## FAQ

<AccordionGroup>
  <Accordion title="Can I override a Block verdict?">
    Yes. GitHub repository admins can bypass branch protection and force-merge. CyBox logs the original verdict ‚Äî the override decision stays with your team.
  </Accordion>
  <Accordion title="What if no rules match?">
    The PR is Allowed by default. The GitHub Check passes and no action is taken.
  </Accordion>
  <Accordion title="Does PR Firewall scan the full repo or just the diff?">
    Conditions based on findings (Severity, Secret detected, Trust boundary risk) evaluate **introduced findings only** ‚Äî those mapped to the changed lines in the PR. Conditions like Path and File match against the set of changed files regardless of findings.
  </Accordion>
  <Accordion title="What if the scan fails?">
    If the CyBox scan doesn't complete, a neutral check is posted. The PR is not blocked. You can retry manually from the asset detail page.
  </Accordion>
  <Accordion title="Can I scope a rule to one repository?">
    Yes. In the rule builder, set the Scope dropdown to specific repositories instead of the default "All Protected".
  </Accordion>
</AccordionGroup>
