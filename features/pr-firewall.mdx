## Where it sits

Most teams have no control layer between AI-generated code and production. A developer accepts a Copilot suggestion, opens a PR, it merges, it ships. PR Firewall closes that gap.

It sits **between the pull request and the merge button** — every PR is scanned, evaluated against your rules, and receives a verdict before anyone can merge.

## Verdicts

Rules evaluate top-to-bottom. The **first rule that matches** sets the verdict. If nothing matches, the PR is Allowed by default.

| Verdict | GitHub Check | Effect |
|---------|-------------|--------|
| ✅ **Allow** | Passes | PR can merge — no action required |
| ⚠️ **Warn** | Passes with warning | Developer notified; merge still allowed |
| 🚫 **Block** | Fails | Merge prevented until resolved or admin override |

<Note>
If the CyBox scan itself fails, a neutral check is posted and the PR is **not** blocked. A scan failure never counts as a security verdict.
</Note>

## Firewall Rules

Rules live at **PR Firewall → Firewall Rules**. Drag the grip (⠿) on any row to reorder priority.

### Rule anatomy

- **Name** — shown in the PR Firewall events table
- **Developer Message** — posted in the GitHub PR review when the rule matches. Make it actionable
- **Scope** — which repositories this rule applies to. Defaults to all connected repos
- **Status** — toggle on/off without deleting

---

### Conditions (WHEN)

All conditions in a rule combine with **AND**. Multiple values inside one condition use **OR**.

<AccordionGroup>
  <Accordion title="Path — match changed file paths" icon="folder">
    Triggers when the PR includes changes to files under a specific path pattern.

    **Operator:** `matches` · **Value:** glob — e.g. `/auth/**`, `/api/payments/**`

    | Example | Action |
    |---------|--------|
    | `path: /auth/**` | Block — protected authentication area |
    | `path: /migrations/**` | Block — database migration review required |
    | `path: /config/**` | Warn — flag configuration changes |
  </Accordion>

  <Accordion title="File — match specific filenames" icon="file">
    Triggers when the PR modifies a specific file or pattern.

    **Operator:** `matches` · **Value:** glob — e.g. `*.env`, `Dockerfile`

    | Example | Action |
    |---------|--------|
    | `file: *.env` | Block |
    | `file: Dockerfile` | Warn |
    | `file: package-lock.json` | Warn |
  </Accordion>

  <Accordion title="Code area — match by functional area" icon="code">
    Triggers based on what type of code changed — detected from the diff content, not just the file path. A payment function added to a utility file still matches `payments`.

    **Values:** `infrastructure` · `authentication` · `payments` · `database`

    | Example | Action |
    |---------|--------|
    | `code area: authentication` | Block |
    | `code area: payments` | Block |
    | `code area: infrastructure` | Warn |
  </Accordion>

  <Accordion title="Severity — match by finding severity" icon="shield-halved">
    Triggers when CyBox detected findings at or above the specified severity **in the PR diff** — introduced findings only, not pre-existing issues.

    **Values:** Critical · High · Medium · Low

    | Example | Action |
    |---------|--------|
    | `severity: Critical` | Block |
    | `severity: High` | Block |
    | `severity: Medium` | Warn |
  </Accordion>

  <Accordion title="AI involvement — match AI-generated PRs" icon="robot">
    Triggers when CyBox detected AI tool involvement in the PR.

    **Values:** `present` / `not present`

    Detection signals: Copilot co-author commit metadata · AI config files (`.cursor/`, `CLAUDE.md`, `.windsurfrules`) · AI-assisted PR labels · bot-authored PRs from v0, Lovable, Bolt, Devin.

    | Example | Action |
    |---------|--------|
    | `AI involvement: present` | Warn — require human review |
    | `AI involvement: present` + `severity: High` | Block |
  </Accordion>

  <Accordion title="Secret detected — match exposed credentials" icon="key">
    Triggers when CyBox detected an exposed API key, token, password, or private key in the diff.

    **Values:** `Yes` / `No`

    This is the most critical rule to have active. Secrets committed to code — even briefly — are compromised.

    `secret detected: Yes` → **Block** — always.
  </Accordion>

  <Accordion title="Trust boundary risk" icon="shield">
    Triggers when untrusted input flows toward a privileged operation without proper validation.

    **Values:** `Yes` / `No`
  </Accordion>

  <Accordion title="Dependency change risk — supply chain changes" icon="box">
    Triggers when the PR adds, upgrades, or removes a dependency — regardless of whether a CVE is known.

    **Values:** `Yes` / `No`

    | Example | Action |
    |---------|--------|
    | `dependency change risk: Yes` | Warn |
    | `dependency change risk: Yes` + `severity: High` | Block |
  </Accordion>

  <Accordion title="Documentation-only PR — fast-track docs" icon="book">
    Triggers when every changed file is a documentation file with no code changes.

    **Values:** `Yes` / `No`

    Use this as an **Allow** rule at the top of your stack to exempt docs PRs from stricter Block rules below.
  </Accordion>
</AccordionGroup>

## Starter rule set

Add these rules in order — priority matters:

<Steps>
  <Step title="Allow docs-only PRs">
    `Documentation-only PR: Yes` → **Allow**

    Fast-track documentation changes past all security checks.
  </Step>
  <Step title="Block exposed secrets">
    `Secret detected: Yes` → **Block** · Message: `Secret or credential detected in this PR`

    The most critical rule — never let a credential reach main.
  </Step>
  <Step title="Block auth path changes">
    `Path: /auth/**` → **Block** · Message: `Protected authentication area — security review required`
  </Step>
  <Step title="Block critical findings">
    `Severity: Critical` → **Block**
  </Step>
  <Step title="Warn on AI-generated code">
    `AI involvement: present` → **Warn** · Message: `AI-generated code detected — please review carefully`
  </Step>
  <Step title="Warn on dependency changes">
    `Dependency change risk: Yes` → **Warn** · Message: `Dependency changes introduced — verify supply chain`
  </Step>
</Steps>

## Setup

<Steps>
  <Step title="Install the CyBox GitHub App">
    See [GitHub Integration](/integrations/github) for full instructions.
  </Step>
  <Step title="Enable branch protection">
    Go to your repo → **Settings → Branches → Add branch protection rule**

    Enable **Require status checks to pass before merging** and select **CyBox Security Scan**.
  </Step>
  <Step title="Configure your rules">
    Go to **PR Firewall → Firewall Rules → + Add Rule** and build out your policy.
  </Step>
</Steps>

## FAQ

<AccordionGroup>
  <Accordion title="Can I override a Block verdict?">
    Yes. GitHub repo admins can bypass branch protection and force-merge. CyBox logs the original verdict — the override stays with your team.
  </Accordion>
  <Accordion title="What if no rules match?">
    The PR is Allowed by default. The GitHub Check passes and no action is taken.
  </Accordion>
  <Accordion title="Full repo scan or diff only?">
    Finding-based conditions (Severity, Secret detected, Trust boundary risk) evaluate **introduced findings only** — mapped to the changed lines. Path and File conditions match the full set of changed files regardless of findings.
  </Accordion>
  <Accordion title="Can I scope a rule to one repository?">
    Yes. Set the Scope dropdown in the rule builder to specific repositories instead of "All Protected".
  </Accordion>
</AccordionGroup>
