# Findings and security posture

CyBox is designed to help teams work from signal, not noise.

## What a finding includes

Each finding is a tracked security issue tied to an asset or endpoint.

Typical finding context includes:

- Clear title
- Severity
- Asset or repository
- File location or URL endpoint
- Evidence
- Remediation guidance
- Status
- AutoFix eligibility, where supported

## Severity model

| Severity | What it means |
|---|---|
| **Critical** | Immediate attention required |
| **High** | Serious risk that should be addressed quickly |
| **Medium** | Important, but often manageable in the next cycle |
| **Low** | Lower urgency issue |
| **Informational** | Useful context with little direct risk |

## Status model

| Status | Meaning |
|---|---|
| **Open** | Still active and unresolved |
| **Fixed** | No longer detected after remediation |
| **Ignored** | Dismissed intentionally with reason |

## Triage flow that works well

1. Start with Critical and High
2. Group by asset to spot concentration of risk
3. Review introduced issues separately from historic backlog
4. Use AutoFix where available
5. Ignore only with a clear reason

## Posture over time

CyBox should help teams answer more than just **how many findings exist**.

It should also help answer:

- Is risk getting better or worse?
- Which assets are consistently problematic?
- Are new pull requests introducing issues faster than teams are closing them?
- Where can AutoFix reduce backlog fastest?

## Best practice

Use findings for day-to-day remediation, and use posture views for trend and ownership conversations.
