# AutoFix

AutoFix helps teams move from finding to fix faster.

Instead of stopping at remediation advice, CyBox can generate a pull request for supported findings so engineers can review the proposed change and decide whether to merge it.

## What AutoFix is good for

AutoFix is especially useful when:

- The issue is clear and well-scoped
- The remediation path is straightforward
- Teams want a faster first draft of the fix
- Security wants to reduce backlog without creating more manual work

## The AutoFix flow

<Steps>
  <Step title="Open the finding">
    Review the issue, evidence, and remediation context.
  </Step>
  <Step title="Generate the fix">
    Trigger AutoFix for a supported finding.
  </Step>
  <Step title="Review the pull request">
    Inspect the proposed code change in GitHub like any normal PR.
  </Step>
  <Step title="Merge with confidence">
    Merge after validation, or edit the proposed code if adjustments are needed.
  </Step>
</Steps>

## What to verify before merge

Every AutoFix PR should be reviewed for:

- Correctness
- Side effects
- Fit with local coding patterns
- Test coverage where relevant

## AutoFix and PR Firewall

AutoFix does not bypass governance.

If the generated pull request still introduces meaningful risk, PR Firewall can still warn on it or block it like any other pull request.

## Good rollout advice

Start with one repository and a narrow set of supported finding types. Review a few generated PRs closely, then expand usage once the team is comfortable.
